# Configuration

Complete reference for tube configuration options.

## Configuration File

tube stores its configuration at `~/.tube/config.yaml` by default.

You can specify a custom location using the `TUBE_CONFIG` environment variable:

```bash
TUBE_CONFIG=/path/to/config.yaml tube list
```

## Full Configuration Example

!!! warning "Cloudflare Tunnel is not yet implemented"
    The `domain`, `tunnel_prefix`, and `tunnel:` fields below describe a
    planned feature. They round-trip through the config file but have no
    runtime effect. See [Architecture › Planned: Cloudflare Tunnel](architecture.md#planned-cloudflare-tunnel)
    for the intended design.

```yaml
# Domain settings (reserved for planned Cloudflare tunnel support)
domain: example.com
tunnel_prefix: dev-

# Directory locations
directories:
  config: ~/.tube            # Where generated nginx/dnsmasq configs are written
  logs: ~/.tube/logs         # Service logs
  ssl: ~/.tube/ssl           # SSL certificates
  pids: ~/.tube/pids         # PID files

# Local proxy settings
proxy:
  local_domain: .test        # TLD for local domains
  dashboard_port: 3249       # Dashboard web UI port
  docs_port: 3250            # Documentation port (future)

# nginx configuration
nginx:
  binary: nginx              # Path to nginx binary
  http_port: 80              # HTTP port
  https_port: 443            # HTTPS port

# dnsmasq configuration
dnsmasq:
  binary: dnsmasq            # Path to dnsmasq binary
  port: 53                   # DNS port

# Cloudflare Tunnel settings (not yet implemented)
tunnel:
  enabled: false
  binary: cloudflared
  name: tube-tunnel

# SSL/TLS settings
ssl:
  enabled: true
  auto_https: true
  mkcert_binary: mkcert
  cert_file: ~/.tube/ssl/wildcard.test.pem
  key_file: ~/.tube/ssl/wildcard.test-key.pem
  ca_installed: true

# Dashboard settings
dashboard:
  enabled: true
  port: 3249
  auth:
    username: ""             # Leave empty to disable auth
    password_hash: ""

# Health monitoring (future)
health:
  enabled: false
  interval: 30s
  auto_restart: false

# Logging
logging:
  level: info                # debug, info, warn, error
  format: text               # text or json

# Your projects
projects:
  frontend: 3000
  api: 8080
  admin: 4200
```

### Directories

```yaml
directories:
  config: ~/.tube
  logs: ~/.tube/logs
  ssl: ~/.tube/ssl
  pids: ~/.tube/pids
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `config` | string | `~/.tube` | Where generated nginx/dnsmasq configs are written |
| `logs` | string | `~/.tube/logs` | Service log files |
| `ssl` | string | `~/.tube/ssl` | SSL certificates generated by mkcert |
| `pids` | string | `~/.tube/pids` | PID files for running services |

**Directory Contents:**

```
~/.tube/
├── config.yaml          # Main configuration
├── nginx.conf           # Generated nginx config
├── projects.conf        # Per-project server blocks
├── dnsmasq.conf         # Generated dnsmasq config
├── logs/
│   ├── nginx-access.log
│   ├── nginx-error.log
│   └── dnsmasq.log
├── ssl/
│   ├── wildcard.test.pem      # SSL certificate
│   └── wildcard.test-key.pem  # SSL private key
└── pids/
    ├── nginx.pid
    └── dnsmasq.pid
```

### Proxy Settings

```yaml
proxy:
  local_domain: .test
  dashboard_port: 3249
  docs_port: 3250
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `local_domain` | string | `.test` | TLD for local domains (include the dot) |
| `dashboard_port` | int | `3249` | Port for web dashboard |
| `docs_port` | int | `3250` | Port for documentation server (future) |

**About the Local Domain:**

The `.test` TLD is [reserved by IANA](https://datatracker.ietf.org/doc/html/rfc2606) for testing and won't conflict with real domains. Other safe choices:
- `.test` (recommended)
- `.localhost`
- `.local` (may conflict with mDNS/Bonjour)
- `.dev` (now a real TLD, not recommended)

If you change the local domain, you'll need to run `tube setup` again to update the DNS resolver.

| `binary` | string | `nginx` | Path to nginx executable |

**Finding nginx:**

```yaml

  binary: /opt/homebrew/bin/nginx
```

### SSL Configuration

```yaml
ssl:
  enabled: true
  auto_https: true
  mkcert_binary: mkcert
  cert_file: ~/.tube/ssl/wildcard.test.pem
  key_file: ~/.tube/ssl/wildcard.test-key.pem
  ca_installed: true
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `enabled` | bool | `true` | Enable HTTPS support |
| `auto_https` | bool | `true` | Automatically generate certificates during init |
| `mkcert_binary` | string | `mkcert` | Path to mkcert for local certificates |
| `cert_file` | string | `~/.tube/ssl/wildcard.test.pem` | Path to SSL certificate |
| `key_file` | string | `~/.tube/ssl/wildcard.test-key.pem` | Path to SSL private key |
| `ca_installed` | bool | `false` | Whether mkcert CA is installed in trust store |

**How HTTPS works:**

tube uses [mkcert](https://github.com/FiloSottile/mkcert) to generate locally-trusted certificates:

1. During `tube init`, mkcert's CA is installed to your system trust store
2. A wildcard certificate (`*.test`) is generated
3. nginx serves HTTPS on port 443 with the certificate
4. Browsers trust the certificate without warnings

**Managing SSL:**

```bash
# Check SSL status
tube ssl status

# Regenerate certificates
tube ssl generate --force

# Disable HTTPS (HTTP only)
tube ssl disable

# Re-enable HTTPS
tube ssl enable
```

**Both HTTP and HTTPS are available** - you can access your apps via either protocol.

### Projects

```yaml
projects:
  frontend: 3000
  api: 8080
  admin: 4200
```

Projects are stored as a simple map of name to port. Each entry creates:
- An nginx virtual host routing `<name>.test` to `localhost:<port>`
- A dnsmasq entry resolving `<name>.test` to `127.0.0.1`

**Adding projects via CLI (recommended):**

```bash
tube add myapp 3000
```

**Adding projects manually:**

Edit `~/.tube/config.yaml`:

```yaml
projects:
  myapp: 3000
  another: 4000
```

Then restart services:

```bash
tube restart
```

## Environment Variables

| Variable | Description | Default |
|----------|-------------|---------|
| `TUBE_CONFIG` | Path to configuration file | `~/.tube/config.yaml` |
| `TUBE_<KEY>` | Override any config key for the current process. Use uppercase and replace `.` with `_` (e.g. `TUBE_PROXY_DASHBOARD_PORT`). | (per-key default) |

**Examples:**

```bash
# Use a different config file
TUBE_CONFIG=/path/to/config.yaml tube list

# Override the dashboard port for one run
TUBE_PROXY_DASHBOARD_PORT=3300 tube status

# Combine
TUBE_CONFIG=./test-config.yaml TUBE_PROXY_LOCAL_DOMAIN=.local tube doctor
```

## Configuration Validation

tube validates configuration on load:

- `tunnel_prefix` cannot be empty
- Port numbers must be 1-65535
- Required directories are created automatically
- Invalid project names are rejected

**Validation rules for project names:**
- 1-63 characters long
- Lowercase letters, numbers, and hyphens only
- Cannot start or end with a hyphen
- Cannot be reserved words (e.g., "www", "mail")

## Generated Configurations

tube generates nginx and dnsmasq configurations in `~/.tube/`:

### nginx.conf

```nginx
worker_processes auto;
error_log /Users/you/.tube/logs/nginx-error.log;
pid /Users/you/.tube/pids/nginx.pid;

events {
    worker_connections 1024;
}

http {
    access_log /Users/you/.tube/logs/nginx-access.log;

    # SSL configuration (shared)
    ssl_certificate /Users/you/.tube/ssl/wildcard.test.pem;
    ssl_certificate_key /Users/you/.tube/ssl/wildcard.test-key.pem;
    ssl_protocols TLSv1.2 TLSv1.3;

    # Include per-project server blocks
    include /Users/you/.tube/configs/projects.conf;
}
```

### projects.conf

```nginx
# frontend.test -> localhost:3000
server {
    listen 80;
    server_name frontend.test;
    location / {
        proxy_pass http://127.0.0.1:3000;
        # ... proxy headers
    }
}

server {
    listen 443 ssl;
    server_name frontend.test;
    location / {
        proxy_pass http://127.0.0.1:3000;
        # ... proxy headers
    }
}
```

### dnsmasq.conf

```
# tube dnsmasq configuration
port=53
listen-address=127.0.0.1
no-resolv
address=/.test/127.0.0.1
```

These files are regenerated each time you start or restart services.
